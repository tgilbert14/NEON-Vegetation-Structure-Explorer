# Deploy & landing-page runbook

This app follows the Desert Data Labs NEON-suite flow: a hosted Shiny app on
**Posit Connect Cloud** (deployed straight from this GitHub repo via `manifest.json`),
plus a **GitHub Pages** landing page (`docs/`) with an og social card and a cold-start
pre-warm — mirroring the Small Mammal Tracker / Plant Diversity / Breeding Birds siblings.

```
  Posit Connect Cloud                         GitHub Pages  (docs/)
  └─ runs global/ui/server + R/ + www/        └─ landing page → "Launch" → app URL
     + data/*.rds indexes                        + og:image social card
     + data/sites/*.rds bundles                  + fetch() pre-warm ping (cold-start fix)
```

The landing page is [`docs/index.html`](docs/index.html); the social card is
[`docs/og-image.png`](docs/og-image.png) (regenerate with
`Rscript scripts/make_og_image.R`). After you deploy the app, set the app URL in **one
place** — the `APP_URL` constant near the bottom of `docs/index.html` — then (re)publish Pages.

---

## 1 — Posit Connect Cloud (the host)

Connect Cloud deploys an R app straight from a **public** GitHub repo using `manifest.json`
(generated by [`scripts/write_manifest.R`](scripts/write_manifest.R)).

The manifest is **lean and bundle-only**: it lists `global/ui/server.R`, `R/`, `www/`, the
precomputed `data/*.rds` indexes, the `data/sites/*.rds` bundles, and the demo sample — and
**91 CRAN packages, NOT `neonUtilities`.** `neonUtilities` is referenced by a *computed* name
in `global.R` (`.NEON_PKG`) precisely so the dependency scanner never pins it: a live NEON pull
on a cold free-tier worker is a hang risk. The deployed app runs **entirely from the bundled
`.rds` files**; the optional live-fetch toggle only appears where `neonUtilities` is actually
installed (local dev). Force bundle-only anywhere with `VST_LIVE=0`.

1. Sign in at <https://connect.posit.cloud> with GitHub.
2. **New Content → Shiny → from Git**, pick `tgilbert14/NEON-Vegetation-Structure-Explorer`,
   branch `main`. Connect reads `manifest.json` (`appmode: shiny`) and auto-detects the
   multi-file Shiny app (`global.R` + `ui.R` + `server.R`) — there is no single "primary file"
   to choose; `global.R` is sourced automatically.
3. It restores packages and serves. **Auto-republishes on every push to `main`.**
4. Copy the published URL → set `APP_URL` in `docs/index.html` (and the README badge).

Regenerate the manifest whenever runtime dependencies change:
```r
# R with the app's runtime packages installed:
Rscript scripts/write_manifest.R     # writes manifest.json + self-checks neonUtilities is absent
```

**Cold start:** the free tier sleeps. The landing page's pre-warm `fetch(APP_URL)` on load wakes
it while the visitor reads, so the app is usually warm by the time they click Launch.

## 2 — Publish the landing page (GitHub Pages)

1. Repo **Settings → Pages → Source: Deploy from a branch → `main` / `/docs`**.
2. Page goes live at `https://tgilbert14.github.io/NEON-Vegetation-Structure-Explorer/`.
   (`docs/.nojekyll` is present so Pages serves the folder verbatim.)
3. `docs/og-image.png` (1200×630) is generated by `scripts/make_og_image.R`.
4. Add this app to the **"other NEON series apps"** grid on the sibling landing pages, and add
   the siblings here (already cross-linked in `docs/index.html`).

## 3 — Automatic data refresh (keep the bundles current)

[`.github/workflows/refresh-data.yml`](.github/workflows/refresh-data.yml) runs on the **first
Saturday night of each month** (Arizona time, off-peak) — or on demand from the Actions tab. It
re-pulls NEON Vegetation structure for the bundled sites, rebuilds `data/sites/*.rds` + the
`data/site_index.rds`, and opens a **PR** with the refreshed data. Merging it to `main` triggers
Connect Cloud to auto-republish.

Requires one repo secret (**Settings → Secrets and variables → Actions**):
- `NEON_TOKEN` — your NEON API token (optional but recommended; speeds the pull and lifts rate limits).

Manual run: Actions → *Refresh NEON data bundle* → *Run workflow*. Tick **skip_download** to just
rebuild the index from the already-committed bundles (fast test, no NEON pull).

## 4 — Rebuild the bundles locally (when adding sites)

The committed bundles are derived from raw NEON pulls. NEON's `loadByProduct` is unreliable on
Windows R-4.5.x (access violation) but fine on Linux/CI; locally use **R-4.1.1** for the pull:
```r
Rscript scripts/refresh_data.R         # pull + bundle + rebuild index (set NEON_TOKEN in env)
Rscript scripts/write_manifest.R       # only if the file/package set changed
```
Then commit `data/` (+ `manifest.json` if it changed) and push — Connect republishes.

## Long-term option — Shinylive / WebAssembly (static, zero server)

Because the deployed app has **no hard runtime dependency on `neonUtilities`** (read-only bundled
`.rds`), it is a shinylive candidate that could serve from GitHub Pages alongside the landing page.
**Gate: every package needs a wasm binary** — check each at <https://repo.r-wasm.org/> (especially
`leaflet`, `plotly`, `DT`) and weigh the bundle size before committing.
```r
# install.packages("shinylive")
shinylive::export(".", "docs/app")     # then APP_URL = "app/" (relative) — one Pages deploy
```
